# PRD Review Command

사용자가 제공한 PRD 문서를 검토하고 상세한 피드백을 제공하세요.

## 입력 받기

사용자에게 다음을 요청하세요:

1. **PRD 파일 경로** (예: `docs/prd/feature-name.md`)
2. **특별히 검토받고 싶은 사항** (선택사항)
   - 예: "문제 정의가 명확한지", "성공 지표가 적절한지", "기술적 접근이 타당한지"

파일 경로가 제공되지 않았다면 명시적으로 요청하세요.

## 검토 프로세스

### 1. PRD 문서 읽기

Read 도구를 사용해서 PRD 파일을 읽으세요.

### 2. 완성도 체크리스트

다음 필수 섹션들이 포함되어 있는지 확인하세요:

```markdown
## PRD 완성도 체크리스트

### 문제 & 맥락
- [ ] 명확한 문제 정의
- [ ] 사용자 리서치/검증 근거
- [ ] 성공 지표 정의
- [ ] 비즈니스 타당성/기회 규모
- [ ] 현재 솔루션과 한계점

### 전략 & 범위
- [ ] 제품 비전/방향성
- [ ] 목표와 비목표 명시
- [ ] 타겟 사용자/페르소나 정의
- [ ] MVP 범위 명확히 정의
- [ ] 향후 로드맵/단계 언급

### 솔루션
- [ ] 사용자 스토리 또는 유스케이스
- [ ] 주요 사용자 플로우 설명
- [ ] 핵심 기능과 근거
- [ ] 기술적 접근 방식 (high-level)
- [ ] 주요 디자인 결정 사항 문서화

### 실행
- [ ] 성공 지표와 목표치
- [ ] 타임라인과 마일스톤
- [ ] 의존성 파악
- [ ] 리스크와 완화 방안
- [ ] 열린 질문들 문서화

### 이해관계자 정보
- [ ] 이해관계자 식별 (DRI, 검토자, 승인자)
- [ ] 크로스 펑셔널 요구사항 (법무, 보안 등)
- [ ] 런칭 계획/GTM 접근
```

### 3. 품질 평가

#### 문제 정의 명확성
- **✅ 좋은 예**: "고객 지원 담당자가 주당 15시간 이상을 티켓 수동 분류에 소비하여, 고객 40%가 24시간 이상 응답 대기"
- **❌ 나쁜 예**: "고객 지원 프로세스가 비효율적이고 개선이 필요함"

평가 질문:
- 엔지니어가 이것을 읽고 질문 없이 문제를 이해할 수 있는가?
- 테스트 가능할 만큼 구체적인가?
- 솔루션 편향을 피하고 있는가?

#### 솔루션 명확성
- **✅ 좋은 예**: 구체적인 예시가 있는 사용자 플로우, 의사결정 포인트 문서화
- **❌ 나쁜 예**: 맥락이나 근거 없는 기능 목록

평가 질문:
- 디자이너가 이것으로부터 플로우를 바로 작업할 수 있는가?
- 엔지니어가 각 기능의 "왜"를 이해하는가?
- 인수 기준이 명확한가?

#### 범위 명확성
- **✅ 좋은 예**: 근거와 함께 명시적인 "포함 범위"와 "제외 범위" 섹션
- **❌ 나쁜 예**: "포괄적인 솔루션을 만들 것"과 같은 모호한 진술

평가 질문:
- 두 사람이 이것을 읽고 동일한 범위 경계를 그릴 수 있는가?
- 비목표(non-goals)가 목표만큼 명확한가?
- MVP가 향후 단계와 명확히 구분되는가?

### 4. 주요 이슈 식별

#### 🔴 레드 플래그 (반드시 수정)
- **성공 지표 없음**: 이게 제대로 작동하는지 어떻게 알 수 있나?
- **문제 없는 솔루션**: 사용자 니즈 설명 없이 "X를 만들어야 함"
- **모호한 범위**: 팀이 범위가 불명확해서 추정할 수 없음
- **의존성 누락**: 중요한 의존성이 파악되지 않음
- **사용자 검증 없음**: 고객 인풋 없는 순수 추측
- **충돌하는 요구사항**: 내부 모순

#### 🟡 옐로우 플래그 (개선 필요)
- **범위 과다**: 한 번에 너무 많은 문제를 해결하려 함
- **과도한 명세**: 불필요하게 구현 세부사항까지 지시
- **대안 누락**: 다른 접근법을 고려하지 않음
- **약한 지표**: 의미 있는 성과 대신 허영 지표
- **단계별 계획 없음**: 전부-아니면-전무 방식의 딜리버리
- **불명확한 책임**: DRI 없거나 책임이 불명확

#### 🟢 긍정적 신호 (강조할 점)
- 근거가 있는 강력한 문제 정의
- 명확하고 테스트 가능한 성공 지표
- 단계별 계획이 있는 잘 정의된 MVP 범위
- 대안에 대한 사려 깊은 고려
- 사용자 리서치 인용/데이터 포함
- 명확한 의사결정 프레임워크
- 완화 방안과 함께 리스크 인지

## 검토 질문들

### 문제 & 기회
- 문제가 명확히 정의되고 검증되었나?
- 기회의 크기가 설득력 있나?
- 왜 지금이 이것을 해결할 적기인가?

### 솔루션
- 제안된 솔루션이 근본 문제를 해결하나?
- 더 간단한 대안이 고려되었나?
- 어떤 트레이드오프를 했고 왜?

### 범위
- 이것이 적절한 MVP 범위인가?
- 무엇을 제외했고 왜?
- V2/V3에는 무엇이 오나?

### 성공
- 성공 지표가 의미 있나 (허영 지표가 아닌)?
- 목표가 도전적이지만 달성 가능한가?
- 어떻게 측정할 것인가?

### 실행
- 이 PRD가 팀에게 실행 가능한가?
- 가장 큰 리스크는 무엇인가?
- 무엇이 잘못될 수 있나?

### 사용자
- 구체적으로 누구를 위한 것인가?
- 그들이 이것을 원한다는 증거가 있나?
- 가정을 어떻게 검증할 것인가?

## 베스트 프랙티스

1. **구체적으로**: "더 많은 디테일 필요"가 아니라 정확히 어떤 디테일이 빠졌는지
2. **건설적으로**: 비판만이 아니라 개선안 제시
3. **우선순위화**: 반드시 고쳐야 할 것과 좋으면 좋은 것 구분
4. **질문하기**: 가정과 미지의 것들을 표면화
5. **좋은 점 칭찬**: 잘한 부분도 명확히 짚어주기
6. **청중 고려**: 엔지니어, 디자이너, 리더십 모두 PRD를 다르게 읽음
7. **자기 점검**: 너무 규범적이지 않은가? PM이 문서를 소유하게 하라

## 주의사항

사용자가 **특별히 검토받고 싶은 사항**을 명시했다면, 그 부분에 특히 집중하되 다른 중대한 이슈도 놓치지 마세요.

## 검토 완료 후

검토를 마친 후:
1. 전체 평가 (🟢/🟡/🔴)와 그 이유를 명확히 제시
2. 가장 중요한 3가지 액션 아이템 강조
3. 필요시 후속 검토 시점 제안

이제 사용자가 제공한 PRD를 읽고 상세하고 실행 가능한 피드백을 제공하세요.
